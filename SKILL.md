# 纲目.skill

> 纲举目张。抓住纲领，条目自明。

## 何时触发

当用户抛出一个**复杂、模糊、信息多**的问题时启用。

典型场景：
- "帮我分析一下这个供应商要不要合作"
- "这个仲裁案件有什么风险"
- "这个产品方案优先级怎么排"
- "我现在很乱，不知道下一步该做什么"
- 任何超过200字的复杂询问

❌ 不触发：简单问答、技术操作、文件读写、日常确认

**触发词**：
- "怎么选"、"什么方向"、"该怎么办"、"从何下手"
- "什么最重要"、"乱的很"、"帮我想想"

**和执要的分支规则**：
- **同一问题域，纲目和执要不串行**——用户按自身状态选择入口
- 问题混沌、方向未定 → 走纲目
- 方向明确、选项太多 → 走执要
- 唯一例外：纲目输出三个行动后，用户说"都重要怎么排" → 切换到执要重新排序

## 反模式与黑名单

| 反模式 | 为什么不行 | 正确做法 |
|--------|-----------|---------|
| 把纲目当分析工具输出过程 | 纲目的输出是结论+行动，不是分析过程。用户要的是"怎么做"，不是"我怎么想的" | 直接给**一纲三目一排除**的格式化输出。分析过程自己保留，不输出 |
| "纲"写成多个并列判断 | 核心判断一旦超过一句话就变成了"重点列举"，失去了"抓住纲领"的穿透力 | 纲必须是一句"问题的本质是XXX"的判断句。写完自问：如果只能留一句，是哪句？ |
| 三目都是"调查/了解/分析"类 | 三个行动都是软性动作（"调查市场""了解风险""分析可行性"），没有可执行的硬动作 | 至少有一个行动必须是可执行的硬动作（"找张三要数据""发邮件确认"），否则用户看完还知道该干什么 |
| 跳过排除直接输出 | "不做什么"比"做什么"更重要——排除是对方向的二次确认 | 排除是必选项。至少写1条。如果实在没有需要排除的，写"当前没有明显需要排除的事项" |
| 对轻量问题做全格式输出 | 用户问"这个会议时间改不改"也要出一套一纲三目一排除，显得小题大做 | 明显是小问题（判断依据<3句话可说明白）直接回答，不走纲目格式 |

## 输出格式（严格遵循）

对所有输出执行以下格式，不多不少，不加额外分析。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
**纲目.skill — 纲举目张**

**一、纲（核心判断）**
一句话说清问题的本质是什么。只选一个，不列清单。

**二、目（三个关键行动）**

| # | 行动 | 说明 |
|---|------|------|
| 1️⃣ | ★ **打歼灭战** | 先集中力量干一件事，抓住主要矛盾 |
| 2️⃣ | 🔍 **事实确认** | 还缺什么关键信息，必须先查清 |
| 3️⃣ | 🛡 **留退路** | 如果A方案不行，B方案是什么 |

> 💡 三个行动是通用框架，允许按场景调整：如果某个行动在当前问题下不适用，可以缺省，不必硬套。有时两个行动加一个排除，比三个行动更精准。

**三、排除（不做什么）**
✗ 这事不关键，先放着
✗ 这是表象，不是根源
✗ 这是别人的事，我们管不了

───────────────────────────────────────
*追问：以上方向是否抓住了你的痛点？需要我深挖哪一条？如果不需要请回复"过"，如果有遗漏请补充。*

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 执行流程

### 第0步：自洽性校验（输出前自查）

在输出结论前，快速做一次**逻辑自检**：

| 检查项 | 问题 | 通过标准 |
|--------|------|---------|
| 🔄 历史一致性 | 这个判断与之前类似的决策/结论是否有冲突？ | 如有冲突，展开写明矛盾点和选择理由 |
| ⏳ 时间稳定性 | 把结论推演到未来3/6个月，核心假设还成立吗？ | 假设发生变化时，结论是否仍站得住 |
| 🔀 反向推演 | 如果有人完全反对你的结论，最有力的理由是什么？ | 能说出1个有力的反方观点，说明不是盲人摸象 |

- 自检未通过（🔴严重冲突）→ 暂不输出，先澄清
- 自检通过（🟢一致 / 🟡有偏差但可解释）→ 正常输出

> 🔴 **CHECKPOINT**：自检完成后，将自检结论（🟢/🟡/🔴）和初步的核心判断方向告知用户，问"自检结果是[🟢一致/🟡有偏差]。我初步判断问题的核心是『xxx』方向，这个方向对吗？"——如用户说不对，退回；如确认，进入抓纲。

### 第一步：抓纲

一堆信息里，哪个是决定性的？只回答一个问题：
> **"这个问题解不解决，取决于哪一件事？"**

> 🔴 **CHECKPOINT**：抓纲完成后，将纲展示给用户，问"我判断问题的本质是『xxx』，这个抓住了你的痛点吗？"——如用户说不对/偏了，退回重抓纲；如确认，继续列目。

### 第二步：列目

三个行动按优先级排序，每条不超过一行，不用"可能""也许"。

| 次序 | 原则 | 检查标准 |
|------|------|---------|
| 行动1 | 歼灭战 | 这件事做完，局面是否明显改观？ |
| 行动2 | 调查事实 | 缺什么信息让我不敢做决定？ |
| 行动3 | 退路 | 最坏情况发生，我还有什么牌？ |

### 第三步：排除

每问一句：
- **"这件事真的重要吗？"** — 回答"不"的，划掉
- **"这不是根源吧？"** — 回答"对"的，划掉
- **"这在我的控制范围内吗？"** — 回答"不在"的，划掉

> 🔴 **CHECKPOINT**：排除完成后，在正式生成输出前问用户"你想看完整的一纲三目一排除，还是只要纲和第一个行动的简明版？"——用户有输出篇幅选择权。同时确认排除的项是否准确：'我划掉了X、Y、Z，这些你觉得该划掉吗？'

## 异常处理与降级路径

| 故障码 | 异常场景 | 触发条件 | 处理动作 |
|-------|---------|---------|---------|
| **F001** | 问题太模糊，无法抓纲 | 输入信息太少或内容过于笼统，无法回答"这个问题解不解决取决于哪一件事" | 降级至"问题澄清"模式：不输出纲目格式，先输出1-2个澄清性问题帮用户聚焦；用户回答后再重新走抓纲流程 |
| **F002** | 纲判断与用户预期不符 | 输出纲后用户说"不对"/"不是这个"/"偏了" | 立即中止，退回抓纲：将用户的修正意见作为新输入重新判断。不在旧纲上修补 |
| **F003** | 只能列出1-2个关键行动 | 列目时想了3个但其中1-2个明显牵强 | 诚实以2个行动输出，不要硬凑第3个；在输出中标注"当前尚缺第三个行动，如后续发现可补充" |
| **F004** | 自检严重冲突未通过 | Step 0 显示🔴（与历史判断或已知事实矛盾） | 不输出，先向用户呈现矛盾点："我目前的判断与XX存在冲突，需要你先澄清这个问题再继续" |
| **F005** | 用户三次追问都不认同 | 追问机制中用户连续3次说"不对"或修改方向 | 输出"当前信息不足以做纲目判断"并建议先走钩玄做信息梳理，或由用户补充更多上下文后再来 |

> **使用说明**：触发任一异常路径时，在输出末尾附加 `⚠️ 降级事件: [故障码] - [异常场景]` 记录。例如：`⚠️ 降级事件: F001 - 问题太模糊，已降级至澄清模式`

## 追问机制

输出完成后，附一句追问，引导用户聚焦下一步：
```
*追问：以上方向是否抓住了你的痛点？需要我深挖哪一条？如果不需要请回复"过"，如果有遗漏请补充。*

作用：
- 避免我一口气说完，用户觉得不对却不好意思打断
- 让对话从"我给答案"变成"一起琢磨"
- 用户选择一条深挖时，重新走纲目流程
- 用户回复"过"或确认时，自动切换到执要（如需排序）或笃行（如需执行）

## 使用示例

### 示例1：坏纲目 vs 好纲目（同源输入）

**输入**（EPC项目决策："某城投平台公司发包一个2.8亿的EPC项目，我司产能已饱和，甲方付款有3个月滞后期。接还是不接？"）

**坏纲目**（写成了分析过程）：
> 这个问题的核心是产能不足和付款风险。建议从以下几个方面考虑：第一，我司目前产能已经饱和，再接的话需要增加20%的人员；第二，甲方付款有滞后期，这是很大的现金流风险；第三，这个项目体量2.8亿很大，放弃又可惜。所以建议再评估一下。

**好纲目**（结论先行）：
> **一、纲**：当前决策的实质不是"接不接"，而是"有没有条件接"——产能和现金流都不支撑，除非甲方提供前置条件。
> **二、目**：1️⃣ 打歼灭战 → 要求甲方提供预付款保函或缩短付款账期，作为接盘的前提条件 2️⃣ 事实确认 → 查清城投平台近3年的付款履约记录，了解是否有过超期或赖账 3️⃣ 留退路 → 即使接了这个项目，也要把对外分包比例控制在50%以上，降低自有资金占用
> **三、排除**：✗ 不要因为"2.8亿大单"的诱惑跳过尽调 ✗ 不要先接再谈付款条件

**对比要点**：坏纲目把分析过程当输出，用户看完了还要自己提炼结论；好纲目直接给结论+行动+排除，用户看完知道"先做什么、查什么、防什么"。

### 示例2：轻量问题（不走纲目格式）

**输入**："这个会议时间改不改？"

**正确做法**（不走纲目格式，直接回答）：
> 改。理由：1）原时间与住建局的季度汇报冲突，关键参会人无法出席；2）改到周五下午后与会率和议题完成度都更高。建议今天就发变更通知。

### 常见避坑

| 场景 | 容易犯的错 | 怎么避 |
|------|-----------|--------|
| 用户提供的背景信息量大但混杂 | 急于输出结论，把某个局部信息当成了全貌的"纲" | 先问1-2个澄清问题确认关键事实（"你说的X和Y之间有关系吗？"），再做判断 |
| 三目都是内部动作（"开会讨论""内部培训"） | 列出的行动只有内部消耗，没有对外部问题的直接影响 | 至少有一个行动必须是直接作用于外部问题的动作（"给对方发函""找供应商谈"），而非内部事务 |
| 排除项写成了"未来计划" | 排除写了"不着急做""以后再说"，没有给出真正不再做的理由 | 排除的标准是"在当前局面下做这件事是错的或无效的"，不是"现在不做以后再做" |
| 用户反馈"全都对，但一个也没法做" | 纲判断正确、三目方向也对，但每个行动都超出了用户的能力边界 | 如果用户反馈"做不到"，做二次降级：把三目改为"你能做的"版本，保持原则不变但降低动作尺度 |

## 在绳墨生态中的位置

绳墨是思维工具链的组合入口（参见 `productivity/shengmo`）。本 skill 是链中的 **第二环（方向判断）**：

```
钩玄(提炼) → 纲目/执要(判断) → 拷问(验证) → 笃行(执行)
                ↑
             你在这里
```

- **上游**：钩玄输出的信息骨架（核/据/疑/用）→ 作为纲目判断的上下文
- **与执要的关系**：同一问题域**二选一**。状态混沌走纲目，方向明确选项多走执要。唯一例外：纲目输出三个行动后用户需要排序 → 切换执要接手
- **下游**：纲目的输出可直接进笃行（简单执行），也可先进拷问验证再进笃行
- **独立调用**：信息已清楚、无需钩玄提炼 → 直接调用纲目
- **接入绳墨**：通过绳墨入口调用时，自动从钩玄接过上下文

## 风格要求

- 每条行动不超过一行
- 不用"可能""也许""建议"
- 直接给结论，不铺垫
- 整体输出不超过手机一屏
- 不输出分析过程，只输出结果
